# Claude Code 整体架构总览：把整个系统连起来看

如果把现有那些专题文档一篇篇读完，你大致已经能拼出 Claude Code 的内部结构。但“知道各个部分”不等于“理解整个系统怎么协同工作”。这篇总览的目标，就是把那个缺口补上。

它不是逐篇复述，而是试图回答一个更核心的问题：

Claude Code 作为一个生产级 AI 编程代理，到底是怎么被设计出来、又是怎么跑起来的？

我会把它拆成几个层面来看：

1. 入口与会话启动
2. 系统提示词与环境注入
3. 主代理循环
4. 工具运行时与权限边界
5. 上下文管理与压缩
6. 状态持久化与记忆系统
7. MCP 与外部集成
8. 远程控制面与整体取舍

这篇的重点不是细节穷举，而是把各子系统放回一个统一架构里。

---

## 1. 先把 Claude Code 看成什么

很多人会把 Claude Code 理解成：

“一个终端里的聊天助手，外加一些工具调用能力。”

从源码看，这个理解偏浅。更准确的说法是：

Claude Code 是一个**面向长会话、带权限边界、可调用工具、可持久化状态、可接外部系统的代理运行时**，CLI 只是它最显眼的交互壳。

这一区别很关键。

因为如果你把它当“聊天工具”，很多复杂度会显得莫名其妙；但如果你把它当“代理运行时”，很多设计就都说得通了。

从整体上看，它至少包含这几层：

1. 入口层
2. 提示词装配层
3. 主循环层
4. 工具执行层
5. 权限与安全层
6. 上下文管理层
7. 持久化与记忆层
8. 外部扩展与控制面

Claude Code 的复杂，恰恰来自这些层都是真的，不是装饰。

---

## 2. 入口层：不是简单把用户输入扔给模型

在表面上，Claude Code 的入口像是：

- 用户在 CLI 里输入一句话
- 系统调用模型
- 再把回答打印回来

但实际启动阶段已经做了很多事：

- 确定工作目录和 project root
- 建立 session identity
- 初始化全局状态和响应式状态
- 决定 permission mode
- 加载本地设置和项目设置
- 建立 telemetry / logging 相关对象
- 初始化 UI 或 SDK 观察链路
- 准备工具池、命令上下文和集成点

换句话说，**第一轮模型调用之前，Claude Code 已经是一个完整运行时了**。  
模型不是入口本身，而是运行时里最重要的一个执行参与者。

---

## 3. 系统提示词装配层：真正的前门其实在这里

Claude Code 给模型的“系统提示词”不是一个静态文本文件，而是一条装配线。

里面会合成：

- 身份与安全约束
- 编码行为规范
- 工具说明
- 环境信息
- CLAUDE.md / memory 内容
- 会话级附加指导
- MCP 说明
- 模型补丁
- 语言与输出风格

如果你从架构角度看，这层实际上在做五件事：

1. **定义策略**  
   模型应该怎么做事，什么该拒绝，什么该谨慎。

2. **解释运行时契约**  
   工具怎么用、权限怎么触发、系统提醒如何理解、压缩后会发生什么。

3. **注入本地规则**  
   CLAUDE.md 和 memory 会影响模型的长期行为。

4. **修补模型缺陷**  
   很多 prompt patch 明显是在给具体模型代打行为补丁。

5. **服务缓存与性能**  
   静态段和动态段被刻意分开，是为了 prompt cache。

这就是为什么我会说：Claude Code 的真正前门不是 CLI，而是提示词装配层。  
因为模型看到的世界、理解的工具和遵循的规则，都是在这里被塑造出来的。

---

## 4. 主代理循环：整个系统的操作核心

一旦 runtime 和 prompt 准备完成，系统进入主查询循环。

这个循环负责：

- 接收用户消息
- 调用模型
- 流式消费输出
- 识别工具调用
- 执行工具
- 回收工具结果
- 处理错误恢复
- 管理上下文压力
- 决定进入下一轮还是终止

这里最关键的设计不是“有个 while 循环”，而是 Claude Code 把整个过程实现成了一个**流式状态机**。

这点非常重要，因为生产级 coding agent 要面对的不是普通问答场景，而是：

- 模型输出可能中断
- 工具调用可能在流中途出现
- 用户可能 abort
- prompt 可能超长
- 主模型可能失败后切 fallback
- 工具结果可能需要边跑边回流

如果只用简单的 request/response 思维，这套系统根本接不住这些情况。

也因此，`query.ts` 那个巨型文件虽然难看，但确实承担了真实的“中枢神经”角色。

---

## 5. 工具运行时：Claude Code 本质上是工具驱动代理

从产品表面看，Claude Code 是个“写代码的 AI”。  
从运行时表面看，它更像一个**由模型驱动的工具执行系统**。

工具层至少提供了这些能力：

- 统一工具契约
- 输入 schema 校验
- 面向模型的 prompt 描述
- 权限接入
- 并发属性声明
- 结果映射
- 大结果落盘
- 延迟加载和过滤

这一点很关键。Claude Code 的工具不是附属功能，而是主循环里的一等公民。

### 这里真正的设计亮点

最值得学的不是工具数量，而是每个工具都必须描述自己：

- 接受什么输入
- 是否只读
- 是否可并发
- 如何做权限检查
- 如何向模型解释自己
- 如何把结果返回到 conversation

一旦这些都统一，Claude Code 就不是“挂了一堆函数”，而是有了真正的工具运行时。

### Streaming tool execution 改变了产品体验

启用流式工具执行后，系统不会等 assistant 全部说完再跑工具，而是：

- 流里一旦发现 `tool_use`
- 就立刻交给执行器
- 并发安全的工具还能同时跑
- 已完成的结果还能在 assistant 还没说完时提前回流

这看起来像“体验优化”，但本质上是架构设计。  
它直接决定了 Claude Code 在用户眼里到底像不像一个“反应快、持续工作的代理”。

---

## 6. 权限与安全层：真正把它从玩具变成产品的边界

没有权限系统，Claude Code 只是一个拥有本地执行能力的模型。  
那不是产品，那是风险放大器。

Claude Code 的安全边界由多层叠起来：

- permission mode
- allow / deny rules
- 工具级权限逻辑
- 路径校验
- Bash 命令解析与只读识别
- sandbox 适配
- 某些模式下的 classifier 辅助判定

最关键的一点是：它没有幻想“一个总开关解决所有安全问题”。  
相反，它承认不同表面的风险结构完全不同：

- 文件读取不是 Bash
- Bash 不是 MCP tool
- MCP tool 不是 CLAUDE.md
- 只读操作和写操作也不是一回事

所以安全策略必须分层。

### 为什么 Bash 被单独重点对待

因为 shell 是最开放、最难静态约束、也是最容易出事的表面。  
所以 BashTool 那条路径特别深：

- 命令解析
- compound command 拆分
- substitution 检查
- 只读识别
- 路径校验
- sandbox 判断
- 特殊命令拦截

这不是洁癖，而是让“AI + shell”勉强可用的最低成本。

---

## 7. 上下文管理层：Claude Code 假定长会话是常态

很多系统把长会话当例外，Claude Code 明显把它当常态。

这会直接影响架构：

- token 要持续估算
- 历史不能无脑累积
- tool result 不能无限留着
- prompt cache 不能随便打穿
- compact 失败还得有恢复策略

所以你会在很多地方看到上下文管理的痕迹：

- token estimation
- pre-request pruning
- snip / microcompact
- reactive compact
- auto compact
- session memory compact
- post-compact cleanup

在很多 AI 产品里，上下文处理是个工具函数；在 Claude Code 里，它是核心子系统。

### 这背后的正确心智模型

Claude Code 并不把 context window 当“固定容量”，而是把它当一项**持续波动、需要预算和调度的运行时资源**。

对长会话代理来说，这正是最合理的思路。

---

## 8. 持久化与记忆层：Claude Code 默认会话应该能活下去

Claude Code 持久化的东西比一般用户想象得多：

- transcript
- session metadata
- permissions
- settings
- CLAUDE.md 规则
- memory 文件
- compact boundary
- resume / fork 上下文

这意味着它不是把自己当短命聊天室，而是当“工作环境”。

### 这里其实有两类不同的 memory

#### 指令型记忆

例如：

- CLAUDE.md
- rules
- local instructions

它们的作用是影响模型未来怎么做事。

#### 运行型记忆

例如：

- transcript
- extracted memory
- session persistence

它们的作用是让系统跨时间延续工作。

这两类记忆不能混为一谈。  
前者是行为塑形，后者是会话连续性。

---

## 9. MCP 与外部集成层：Claude Code 正在变成平台

MCP 这部分透露出一个很重要的方向：Claude Code 不只是本地工具合集，而是在往**代理平台**演化。

通过 MCP，它可以：

- 发现外部工具
- 支持多种 transport
- 跑 OAuth
- 动态刷新工具和资源列表
- 把外部能力注入模型工具空间

这意味着：

- 内建工具不再是能力边界的全部
- 产品能力可以按环境动态扩展
- Claude Code 运行时正在从“单体应用”走向“可扩展平台”

这不是小功能，而是架构方向。

---

## 10. 远程设置、feature flag 与控制面

前面的文档也能看出来，Claude Code 的行为不只由本地代码决定。还有多层控制面：

- 静态代码
- 系统提示词
- 本地配置
- 项目配置
- managed config
- feature flag
- remote settings

这让系统更灵活，但也更难一眼看懂。  
同一段行为可能因为下面任一因素变化：

- 代码版本
- prompt patch
- feature gate
- 远程 flag
- 企业管理员设置
- MCP server 接入状态

这正是现代 SaaS 式代理产品的典型特征。

## 11. Claude Code 这套架构到底在优化什么

把整个代码库串起来看，我觉得它主要优化的是下面几件事：

### 1. 真实世界鲁棒性，高于架构洁癖

Claude Code 经常选择“能恢复、能继续跑”的方案，而不是理论上最优雅的抽象。

### 2. 工具使用效率，高于纯聊天体验

这个系统的核心不是把回复说漂亮，而是把工具链跑顺。

### 3. 长会话连续性，高于短轮简单性

持久化、compact、memory 的存在，都是因为它假定会话会持续很久。

### 4. 运营控制能力，高于静态行为固定

prompt patch、feature flag、remote settings 都说明它希望行为可调。

### 5. 多层安全边界，高于单一信任机制

不是靠一个 classifier 或一个 sandbox 打天下，而是规则、验证、权限、路径检查、sandbox 一起叠。

这就是一个生产代理系统会有的优化方向。

---

## 12. 复杂度为什么会长成这样

Claude Code 的复杂度不是来自某一个点，而是来自几个难题叠在一起：

- 模型本身不确定
- 工具执行有副作用
- 本地机器权限有风险
- 会话会很长
- 状态要能恢复
- 提示词要持续调教
- 外部集成会动态变化
- UI 和 SDK 还得一起支持

真正麻烦的是这些难题会互相放大。

例如：

- 一旦支持流式工具执行，主循环就更复杂
- 一旦工具能改机器，权限系统就更关键
- 一旦会话可恢复，持久化和 compact 边界就更难做
- 一旦 prompt 动态注入很多内容，缓存和上下文管理压力就更大

这就是 Claude Code 架构真正的故事：  
不是某个子系统有多大，而是所有子系统之间的相互作用，把系统推成了一个完整的生产运行时。

---

## 13. 我的总体判断

Claude Code 最准确的定位不是“带工具的终端聊天助手”，而是：

**一个带终端界面的生产级代理运行时。**

它把下面这些东西放进了一个系统里：

- 提示词控制
- 有状态执行
- 工具运行时
- 权限边界
- 上下文压缩
- 持久化
- 外部扩展
- 远程控制面

所以它既不小，也不优雅得像教科书，但它非常“真实”。  
它更像一个在大量生产环境压力、失败恢复需求、延迟约束和安全边界下，逐渐长成的系统。

也正因为这样，它非常值得研究。
