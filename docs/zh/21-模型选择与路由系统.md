# 模型选择与路由系统

## 概览

Claude Code 在 `src/utils/model/` 下实现了一套复杂的模型选择与路由机制。它要解决的问题不是简单的“当前用哪个模型”，而是把用户输入的抽象意图，例如 `sonnet`、`best`、`opus[1m]`、`opusplan`，稳定映射到不同 API 提供商对应的真实模型标识符，同时还要兼顾订阅等级、能力开关、价格层、子代理继承、上下文窗口和废弃迁移。

这套系统的核心设计是三层分离：

- 用户面向的稳定别名层
- 内部统一的模型注册表
- 运行时按 provider 与上下文解析出的具体模型字符串

## 1. 模型注册表与别名

基础设施在 `configs.ts`。这里把每个模型定义成 `ModelConfig`，分别给出 `firstParty`、`bedrock`、`vertex`、`foundry` 四类 provider 对应的模型 ID。所有配置最终进入 `ALL_MODEL_CONFIGS`，成为运行时解析的单一事实来源。

这层注册表的重要性在于：

- 内部可以用简短 key 管理模型，如 `sonnet46`、`opus46`
- provider 差异被集中封装，不必散落在业务逻辑中

系统还维护了 `CANONICAL_ID_TO_KEY` 的反向映射，把 canonical first-party 模型名映射回内部 key。这样即便用户通过 `modelOverrides` 把某个模型换成私有 profile、ARN 或自定义 provider 字符串，系统后续仍然能回到 canonical 语义层，继续做价格和能力判断。

用户真正接触到的是别名层，主要包括：

- `sonnet`
- `opus`
- `haiku`
- `best`
- `sonnet[1m]`
- `opus[1m]`
- `opusplan`

这些别名本质上是一层版本稳定性抽象。用户写 `sonnet`，并不显式绑定 Sonnet 4.5 或 4.6；当默认版本升级时，alias 会自动指向新代际，而不要求用户手工迁移配置。

## 2. Canonical 名称解析

不同 provider 的模型标识差异很大。first-party 可能是 `claude-sonnet-4-6`，Bedrock 可能是带区域前缀或 inference profile 的形式。为了让后续逻辑不依赖这些表面差异，系统提供了两层 canonical 化：

- `firstPartyNameToCanonical()`：去掉日期、provider 特有后缀和变体
- `getCanonicalName()`：先处理 override，再提取 canonical 名称

匹配采用“更具体版本优先”的顺序，例如先尝试 4.6，再尝试 4.5，再尝试裸 4，避免 `4-6` 被错误匹配到更宽泛的 `4`。

这一步的意义并不只是格式统一，而是为后续所有横切逻辑提供共同语义基础：

- 定价
- 能力开关
- allowlist
- 订阅层展示
- 子代理继承策略

## 3. 主模型选择优先级

主循环模型由 `getMainLoopModel()` 决定，优先级链条很明确：

1. 会话中的 `/model` 临时覆盖
2. 启动参数 `--model`
3. `ANTHROPIC_MODEL` 环境变量
4. 用户保存的设置
5. 系统内建默认值

内建默认值本身还受订阅等级影响：

- Max / Team Premium：默认 Opus 4.6
- 其他常见用户层级：默认 Sonnet 4.6

若启用了 1M 相关合并逻辑，系统还会自动附加 `[1m]`。

这里体现的是一种“局部覆盖优先，全局演进托底”的思路。用户可以在单次会话里覆写模型，但系统仍然保留 alias 自动升级默认代际的能力。

## 4. 多 Provider 解析

`providers.ts` 通过环境变量决定当前使用哪个 API provider：

- `firstParty`
- `bedrock`
- `vertex`
- `foundry`

之后 `modelStrings.ts` 会根据 provider 从统一注册表中抽出对应的模型字符串。

Bedrock 是最特殊的一支。系统会调用 `getBedrockInferenceProfiles()` 获取当前可用 profile，然后把 profile 与 first-party canonical 模型名做匹配，从而自动找出当前区域中最合适的模型标识。这一层不仅影响可用性，还影响：

- 区域驻留
- IAM 兼容性
- 子代理对父代理 region 前缀的继承

如果父线程本身就带有 Bedrock region prefix，子代理在解析模型时会尽量保留这一前缀，防止无声切换到其他区域。

## 5. 模型覆盖与默认代际差异

`modelOverrides` 允许用户按 canonical 模型名重写 provider 对应的真实模型字符串。这在 Bedrock / Vertex 场景尤其重要，因为企业内部很可能使用定制 profile、私有部署名或 ARN，而不是官方默认 ID。

系统会先按 provider 选出默认模型字符串，再叠加用户 override。

不同 provider 的默认模型代际也不完全一致。典型例子是 Sonnet：

- first-party：优先使用更近代的 Sonnet 4.6
- 第三方 provider：可能继续默认 Sonnet 4.5 这样的稳定版本

这背后的工程现实是：第三方平台的模型上线通常滞后于 first-party。如果默认值过于激进，用户在 Bedrock / Vertex 上更容易遇到不存在的模型。

同时，管理员还可以通过：

- `ANTHROPIC_DEFAULT_SONNET_MODEL`
- `ANTHROPIC_DEFAULT_OPUS_MODEL`
- `ANTHROPIC_DEFAULT_HAIKU_MODEL`

直接 pin 住具体版本。

## 6. 模型能力发现

`modelCapabilities.ts` 实现了一套带缓存的能力发现机制。对符合条件的用户，系统会从 Anthropic API 拉取模型列表，并把能力信息缓存到 `~/.claude/cache/model-capabilities.json`。

缓存的能力记录主要包含：

- `id`
- `max_input_tokens`
- `max_tokens`

这份动态信息会覆盖静态默认值，用于更准确地判断：

- 上下文窗口上限
- 最大输出 token
- 某些 beta 或功能是否应启用

匹配策略先精确匹配，再做包含匹配；缓存还会按模型 ID 长度降序排序，以优先命中更具体的模型项。

这说明 Claude Code 并不完全依赖硬编码配置，而是尝试把“模型能力”看作一种可动态更新的元数据。

## 7. 上下文窗口与输出上限

上下文窗口大小并不是单一字段决定，而是按多层顺序解析：

1. `CLAUDE_CODE_MAX_CONTEXT_TOKENS`
2. 模型名是否带 `[1m]`
3. 动态能力缓存
4. beta header
5. 实验开关
6. 静态默认值 200K

因此 `[1m]` 不是展示标签，而是会影响实际运行时上下文限制的有效信号。

输出 token 上限则按模型族区分。Opus 4.6 可以到 128K，Sonnet 4.6 也能到 128K，但默认值更低；更早期模型则逐步下降到 64K、8K 甚至 4K。

系统还引入了保守的默认占位策略，通常先按较低默认值保留输出预算，再在需要时升级。这是典型的生产环境优化：高上限要保留，但大多数请求并不会实际打满。

## 8. 每模型功能开关

Claude Code 中很多能力都是按“模型 + provider”组合判定的，而不是单纯看模型字符串：

- thinking
- adaptive thinking
- interleaved thinking / ISP
- structured outputs
- web search
- context management

常见判断顺序是：

1. 先看 3P capability override
2. 再看静态 allowlist / denylist
3. 最后按 provider 默认策略兜底

例如 thinking 支持上，first-party 与 Foundry 对 Claude 4+ 往往更积极，而 Bedrock / Vertex 则收窄到 Sonnet 4+ / Opus 4+。这意味着“同名模型在不同 provider 上能力等价”这个假设在 Claude Code 里并不成立。

对第三方 provider，还可以通过 `ANTHROPIC_DEFAULT_*_MODEL_SUPPORTED_CAPABILITIES` 一类环境变量声明能力集合，进一步覆盖静态检测。

## 9. 子代理模型选择

`agent.ts` 负责子代理模型解析，优先级大致是：

1. `CLAUDE_CODE_SUBAGENT_MODEL`
2. 工具 frontmatter 指定的 `model`
3. agent 配置里的 `model`
4. 默认 `inherit`

系统在这里做了一个很细的优化：如果子代理配置的是裸 alias，例如 `opus`，而父线程当前模型本来就是同一族，那么子代理会直接继承父线程的精确模型字符串，而不是重新解析 alias。

这样可以避免“看似继承，实际降级”。尤其在第三方 provider 上，重新解析 alias 可能会回到更老的默认模型。

另一个值得注意的点是技能模型覆盖与 1M 语义的联动。如果当前线程是 `opus[1m]`，技能 frontmatter 只写 `model: opus`，系统仍会尽量保留 `[1m]` 后缀，防止上下文窗口无声掉回 200K。

运行时切换上，`opusplan` 会在 plan 模式切到 Opus、执行模式回到 Sonnet；而 `haiku` 在 plan 模式下也会升级到 Sonnet，以保证规划阶段的推理能力。

## 10. 废弃与迁移

`deprecation.ts` 维护废弃模型表，并为不同 provider 记录不同退役日期。这里没有采用统一日期，是因为 Bedrock / Vertex 的上线与退役节奏本来就经常滞后或错位。

对于 first-party 上已经不再直接提供的旧 Opus 4.0 / 4.1 变体，系统还会做静默 remap，把它们导向当前 Opus 默认值。这个行为仅在 first-party 生效，并且可以通过 `CLAUDE_CODE_DISABLE_LEGACY_MODEL_REMAP=1` 关闭。

若第三方 provider 上模型校验失败，系统还会给出“上一代稳定版本”的建议回退：

- `opus-4-6` 回退到 Opus 4.1
- `sonnet-4-6` 回退到 Sonnet 4.5
- `sonnet-4-5` 回退到 Sonnet 4.0

UI 层也会检测“用户 pin 住老版本，但 alias 已经有更新映射”的情况，用提示把用户引导回别名路径。

## 11. 定价层

`modelCost.ts` 为不同模型族维护每百万 token 的价格层。Sonnet 系列采用统一价格，Opus 4 / 4.1、Opus 4.5 / 4.6、Haiku 系列则各有独立层级。

其中最特殊的是 Opus 4.6。它存在标准与 fast mode 两套价格，系统会检查 API 返回的 `usage.speed`：

- 标准速度：`$5 / $25`
- fast mode：`$30 / $150`

也就是说，同一个 canonical 模型名在计费层并不总是对应单一价格。

如果模型未命中价格表，系统会回退到默认主循环模型的成本层，再不行则退到一个保守兜底值，并记录未知模型成本事件。

## 12. Beta Header 与行为配置

Claude Code 对模型行为的调优大量依赖 `betas.ts` 生成的 beta header，而不是只靠提示词层补丁。系统会按模型和 provider 决定是否附加：

- interleaved thinking
- redact thinking
- context management
- structured outputs
- token-efficient tools
- web search
- 1M context
- prompt caching scope

Bedrock 还要额外处理某些 beta 被放在 body params 而不是 HTTP header 的情况。

从实现上看，这些 beta 更像“协议层能力协商”，而不是单纯的 prompt 文案切换。

## 13. Fast Mode 与 Effort

Fast mode 是 first-party 专属能力，只对 Opus 4.6 生效。它会综合判断：

- provider 是否为 first-party
- 组织是否具备资格
- 当前订阅等级
- feature flag
- 当前模型是否支持 fast mode

当 API 返回 429 或 overloaded 时，系统会进入 cooldown 状态，在冷却期内自动回退到标准速度。

Effort 系统则负责控制推理深度，支持：

- `low`
- `medium`
- `high`
- `max`

解析优先级是环境变量、AppState、模型默认值。`max` 只允许 Opus 4.6，其他模型会被自动降级为 `high`。对外部用户而言，`max` 还被限制为 session 级，不允许持久化。

“ultrathink” 则是另一层关键词驱动机制。启用后，相当于先把默认推理深度抬到 `medium`，再允许带有 `ultrathink` 的请求进一步升档。

## 14. Allowlist 系统

`availableModels` 设置实现了模型 allowlist。匹配分三层：

1. 家族别名，例如 `opus`
2. 版本前缀，例如 `opus-4-5`
3. 完整模型 ID

其中有一个很关键的边界条件：如果 allowlist 同时包含 `opus` 和 `opus-4-5`，系统会把更具体的版本前缀视为收窄条件，而不是仍然允许整个 Opus 家族。

`validateModel()` 在真正发网络请求前，会先检查 allowlist，并对所有已知 alias 直接放行。真正的 live validation 则发送一个最小请求，只验证模型是否存在与可调用。

## 15. 按订阅等级展示模型

模型选择器展示内容会因订阅层显著不同：

- Max / Team Premium：默认 Opus 4.6，Sonnet / Haiku 作为降级选项
- Pro / Team Standard / Enterprise：默认 Sonnet 4.6，Opus 作为升级选项
- PAYG first-party：展示更完整模型集，并附带价格信息
- PAYG third-party：默认更保守，通常停在 Sonnet 4.5 一类稳定版本

1M 变体是否展示还要再经过 `checkOpus1mAccess()` / `checkSonnet1mAccess()` 判定，涉及功能开关和额外计费权限。
