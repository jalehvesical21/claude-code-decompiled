# 认证与授权系统

## Claude Code 凭证管理架构分析

Claude Code 的认证体系不是单一路径，而是一套面向多种运行环境的组合架构。它既要支持最直接的 API key，也要支持 claude.ai / Console OAuth，还要兼容 AWS Bedrock、Google Vertex AI、Azure Foundry 这类云 provider，同时处理 Claude Code Remote、Claude Desktop、SSH 场景下的 token 注入、缓存、刷新和隔离。

相关逻辑主要集中在：

- `src/utils/auth.ts`
- `src/services/oauth/`
- `src/services/api/client.ts`
- `src/utils/secureStorage/`

## 1. 认证方式分层

Claude Code 实际支持四大类认证来源：

1. 直接 API key
2. OAuth 登录
3. 通过环境变量或文件描述符注入的 OAuth token
4. 第三方云 provider 凭证

系统通过一条有优先级的解析链决定当前真正采用哪种来源。

### 1.1 API Key

最直接的路径是设置 `ANTHROPIC_API_KEY`。此时系统会绕过 OAuth，直接把 key 作为 `x-api-key` 发送。

`getAnthropicApiKeyWithSource()` 会返回 key 和来源，来源包括：

- `ANTHROPIC_API_KEY`
- `apiKeyHelper`
- `/login managed key`
- `none`

CI 环境里，环境变量 API key 会直接优先生效；而在普通交互环境中，系统要求它先出现在已批准列表里，避免任意注入的环境变量覆盖用户已有认证状态。

### 1.2 OAuth

交互式用户主要走 OAuth 2.0 + PKCE。这里又分成两条：

- **Claude.ai OAuth**：给订阅用户提供 `user:inference`
- **Console OAuth**：给 API 用户提供 `org:create_api_key`，登录后再由后端生成 API key

系统用 `shouldUseClaudeAIAuth()` 判断 scope 里是否包含 `user:inference`，从而区分这两条路径。

### 1.3 环境变量 / FD 注入

托管环境中，token 可能直接从外部注入：

- `CLAUDE_CODE_OAUTH_TOKEN`
- `CLAUDE_CODE_OAUTH_TOKEN_FILE_DESCRIPTOR`

这类 token 被视为 inference-only，通常没有 refresh 能力，因此 `expiresAt` 往往是 `null`。

### 1.4 托管上下文保护

`isManagedOAuthContext()` 用来判断当前是否运行在 Claude Code Remote 或 Claude Desktop 之类的托管环境中。一旦处于这种上下文，系统会主动忽略：

- `apiKeyHelper`
- `ANTHROPIC_API_KEY`
- `ANTHROPIC_AUTH_TOKEN`

这样做是为了防止用户终端中的旧凭证泄漏到本应完全受托管 OAuth 控制的会话里。

### 1.5 Auth 开关判定

`isAnthropicAuthEnabled()` 决定第一方 OAuth 是否启用。以下情况会让它关闭：

- 使用 `--bare`
- 已配置 Bedrock / Vertex / Foundry
- 托管上下文外存在外部 bearer token 或 `apiKeyHelper`
- 托管上下文外存在外部 API key

SSH remote 还有特殊分支：如果设置了 `ANTHROPIC_UNIX_SOCKET`，是否启用 auth 主要取决于 `CLAUDE_CODE_OAUTH_TOKEN`，因为真正的凭证是由本地代理注入的。

## 2. API Key 管理

### 2.1 存储

API key 的存储策略是平台感知的：

- macOS：优先写入系统钥匙串
- 其他平台：退回 `~/.claude.json`

在 macOS 上，`saveApiKey()` 使用 `security -i` 并通过 stdin 传入十六进制编码后的 key，而不是把凭证直接放进命令参数里。这是明确的安全设计，目的是规避凭证出现在进程列表或监控日志中。

### 2.2 格式验证

保存前会先用保守正则验证 key 格式，只允许字母、数字、连字符和下划线。真正的可用性验证则通过 `verifyApiKey()` 请求 Anthropic API，由前端的 `useApiKeyVerification` hook 管理状态：

- `loading`
- `valid`
- `invalid`
- `missing`
- `error`

### 2.3 批准列表与归一化

写入配置时，系统不会保存完整 key，而只保存末尾 20 位。这可以在不长期存储完整密钥的前提下，让系统识别一个通过环境变量注入的 key 是否曾被用户批准。

### 2.4 `apiKeyHelper`

`apiKeyHelper` 允许用户配置 shell 命令，动态输出 API key。它采用的是 SWR 风格缓存：

- 默认 TTL 5 分钟
- 冷缓存时阻塞等待 helper 输出
- 热缓存时先返回旧值，再后台刷新
- 用 epoch 机制避免旧请求覆盖新结果

若 helper 来源于项目设置，还必须先通过 workspace trust。否则恶意仓库可以在用户确认前，通过 `.claude/settings.local.json` 触发任意命令执行。

## 3. OAuth 流程

### 3.1 PKCE 授权码流程

OAuthService 负责整个授权流程。关键密码学原语放在 `crypto.ts` 中：

- `generateCodeVerifier()`
- `generateCodeChallenge()`
- `generateState()`

流程大致是：

1. 启动本地 HTTP 监听器
2. 生成 verifier / challenge / state
3. 构造自动与手动两类授权 URL
4. 打开浏览器
5. `AuthCodeListener` 接收回调
6. 校验 state，防止 CSRF
7. 用 code 交换 token

### 3.2 授权 URL 组装

`buildAuthUrl()` 会拼接：

- `client_id`
- `response_type=code`
- `redirect_uri`
- `scope`
- `code_challenge`
- `code_challenge_method`
- `state`

并支持附加：

- `orgUUID`
- `login_hint`
- `login_method`

系统默认一次性请求所有 scopes，而不是延迟追加。后端则支持在 refresh 阶段扩展 scope，因此旧 token 也可能在后续刷新后拿到新权限。

### 3.3 Token 交换与刷新

`exchangeCodeForTokens()` 负责首次拿 token；`refreshOAuthToken()` 负责后续刷新。

这里有一个明显的规模化优化：如果全局配置和安全存储里已经有足够的 profile 衍生信息，例如 `billingType`、`subscriptionType`、`rateLimitTier`，刷新时就会跳过 `/api/oauth/profile`，以减少大量重复请求。

### 3.4 非交互快速路径

`authLogin` 还支持通过：

- `CLAUDE_CODE_OAUTH_REFRESH_TOKEN`
- `CLAUDE_CODE_OAUTH_SCOPES`

直接跳过浏览器交互流程。这使得 CI/CD 与托管环境能预置刷新 token 完成自动登录。

## 4. 凭证存储层

安全存储采用 strategy + fallback 架构，共三层：

1. macOS Keychain
2. Plain Text Storage
3. Fallback Composite

Plain Text Storage 位置是 `~/.claude/.credentials.json`，权限设置为 `0o600`。Fallback 组合层在 primary 写失败时会退到 secondary，并尽量删除 primary 里的旧记录，防止读取时旧值优先、把新 refresh token 遮蔽掉。

macOS Keychain 还带 stale-while-error 缓存。如果一次读取发生瞬态错误，系统不会立刻把缓存视为 `null`，而是继续使用上一份成功值，避免一次钥匙串读取失败把整个 CLI 都变成“未登录”。

## 5. Remote / CCR 中的 FD 凭证

在 Claude Code Remote 场景里，凭证常通过文件描述符注入。`getCredentialFromFd()` 会先尝试从 pipe FD 读取，然后把读取成功的结果落到 well-known file，供后续子进程和 tmux 边界继续使用。

这种双层设计的原因是：

- FD 自身并不稳定，跨 `exec` 与 pane 边界容易失效
- 环境变量可能继续保留注入提示，但不再带实际内容

凭证会写到类似 `/home/claude/.claude/remote/.oauth_token` 的位置，并使用严格权限：

- 目录 `0o700`
- 文件 `0o600`

并且整个流程只会在 `CLAUDE_CODE_REMOTE` 为真时运行。

## 6. 多 Provider 认证

### 6.1 AWS Bedrock

Bedrock 支持多种来源：

- `AWS_BEARER_TOKEN_BEDROCK`
- `awsCredentialExport`
- `awsAuthRefresh`
- AWS SDK 标准 credential chain

`refreshAndGetAwsCredentials()` 带有 1 小时 TTL，和典型 STS 生命周期一致。预取逻辑 `prefetchAwsCredentialsAndBedRockInfoIfSafe()` 只会在可信工作区提前执行，否则要等 trust 接受之后。

### 6.2 Google Vertex AI

Vertex 使用 `google-auth-library` 的 `GoogleAuth`，scope 是 `cloud-platform`。系统支持：

- `gcpAuthRefresh`
- 短超时 access token 有效性检查
- 1 小时 TTL 缓存

短超时不是偶然，而是为避免在非 GCP 环境里访问 metadata server 时卡死十几秒。

### 6.3 Azure Foundry

Foundry 走两条路径：

- `ANTHROPIC_FOUNDRY_API_KEY`
- `DefaultAzureCredential`

若没有显式 API key，系统会通过 Azure AD 获取 `https://cognitiveservices.azure.com/.default` 范围内的 bearer token。

### 6.4 云凭证 Helper 的安全约束

`awsAuthRefresh`、`awsCredentialExport`、`gcpAuthRefresh` 这些 helper 如果来自项目设置，一样必须在 workspace trust 之后执行。这是和 `apiKeyHelper` 同一类风险控制：不能让仓库配置在用户确认前直接触发命令。

## 7. Token 生命周期与跨进程同步

OAuth token 的生命周期包含：

1. 获取
2. 存储
3. 读取
4. 刷新
5. 失效与清理

### 7.1 过期判断

`isOAuthTokenExpired()` 会提前 5 分钟进入“即将过期”状态。对 `expiresAt === null` 的 token，则永不主动判定过期，因为这类 token 本身没有 refresh 能力。

### 7.2 跨进程缓存失效

多个 Claude Code 进程并发运行时，`invalidateOAuthCacheIfDiskChanged()` 会检查 `.credentials.json` 的 `mtime`。一旦发现另一个进程写入了新 token，就清掉当前进程内存缓存并重新加载。

这个设计处理了一个很典型的场景：终端 A 登录后，终端 B 再登录，服务端可能使 A 的 refresh token 失效。如果 A 永远只信任本地 memoized 缓存，它会长期处于错误状态。

### 7.3 并发刷新去重

刷新逻辑做了三层去重：

1. Promise 级去重
2. 文件锁去重
3. 取锁后重新读取验证，确认是否已被其他进程刷新

拿不到锁时会进行带抖动的有限重试，避免多进程同时刷新同一个 token。

## 8. Auth 状态机

`getAuthTokenSource()` 负责按优先级解析当前 token 来源，顺序大致是：

1. bare mode 中的 `apiKeyHelper`
2. `ANTHROPIC_AUTH_TOKEN`
3. `CLAUDE_CODE_OAUTH_TOKEN`
4. 文件描述符 OAuth token
5. `apiKeyHelper`
6. claude.ai OAuth

它返回带 `source` 与 `hasToken` 的判别联合类型，供 UI 和逻辑层消费。

`authStatus` 再把这些底层来源归纳成用户可见的认证模式：

- `third_party`
- `claude.ai`
- `api_key_helper`
- `oauth_token`
- `api_key`
- `none`

登录流程 `authLogin` 则负责编排：

1. 决定走 claude.ai 还是 Console
2. 检查 refresh token 快速路径
3. 启动浏览器 OAuth
4. 安装 token
5. 拉取 profile、角色和订阅类型
6. 清空相关缓存

订阅类型最终会从 OAuth profile 写入安全存储和全局配置，并向系统其他部分暴露：

- `isMaxSubscriber()`
- `isTeamSubscriber()`
- `isEnterpriseSubscriber()`
- `isProSubscriber()`

企业环境还可以通过 `forceLoginOrgUUID` 强制校验组织归属，并采取 fail-closed 策略。

## 9. 错误恢复

### 9.1 OAuth 401

遇到 401 时，`handleOAuth401Error()` 会：

1. 去重同一失败 access token 的并发处理
2. 清空内存缓存
3. 重新读取安全存储
4. 若磁盘上已存在更新后的 token，则直接恢复
5. 否则再触发刷新

这避免了多条并发 API 请求同时收到 401 后，全部各自独立刷新。

### 9.2 `apiKeyHelper` 失败

SWR 路径下，后台刷新失败会保留旧值并更新其时间戳，防止不停 hammer helper。冷启动失败时还会写入 sentinel，阻止系统错误回退到 OAuth 路径。

### 9.3 Keychain 锁定

macOS 上会检测钥匙串是否锁定，这在 SSH 会话中很常见。系统会缓存锁定状态，避免反复尝试触发同一类失败。

### 9.4 SSL 与网络错误

登录失败输出中会附带 SSL hint，说明 CLI 在错误提示层面区分了证书类失败和普通请求失败。

### 9.5 刷新锁竞争

刷新 token 的文件锁若拿不到，系统会带抖动重试若干次；如果仍失败，则保守返回，让后续 401 恢复路径继续接手。

### 9.6 Logout 顺序

退出登录时，操作顺序是刻意安排的：

1. 先 flush telemetry
2. 再删除 keychain / config 中的 API key
3. 再删除安全存储
4. 再清空 auth、betas、GrowthBook 等缓存
5. 最后重置全局配置

这一顺序避免凭证切换中把残余用户态数据错误归到下一身份上下文里。

### 9.7 自定义 OAuth URL

`CLAUDE_CODE_CUSTOM_OAUTH_URL` 虽然允许替换 OAuth 基地址，但只能命中 allowlist 中的固定端点。否则系统会立即抛错，防止 token 被发往任意地址。
