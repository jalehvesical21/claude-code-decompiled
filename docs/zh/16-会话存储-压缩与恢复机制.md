# Claude Code 的会话存储、压缩边界与恢复机制

一个长时间工作的代理系统，只有在能够停下来、持久化、压缩、再恢复之后依然保持上下文连续性时，才会真正像“工作环境”而不是一次性聊天窗口。

Claude Code 在这件事上明显下了不少功夫。

这篇就专门看它支撑连续性的那条链路：

- transcript 存储
- parent-linked message chain
- sidechain 分支
- compact boundary
- resume / continue / fork session

这块不如 prompt 或 agent loop 那么“好传播”，但它恰恰是产品之所以耐用的重要原因之一。

---

## 1. 会话历史不是扁平日志

一个最朴素的助手会把对话历史存成一个 append-only 数组。Claude Code 不是。

它的 transcript 里，每条消息都会带上链式结构信息：

- message UUID
- parent UUID
- logical parent
- sidechain 状态
- session metadata

这意味着 transcript 不是“过去消息列表”，而是“带结构的会话历史”。

这很重要，因为代理会话本来就不总是一条直线。

用户可能会：

- rewind
- 从旧节点继续
- compact
- fork
- 隔一段时间 resume

扁平日志很难准确表达这些状态变化。

---

## 2. parent-linked chain 解决的是分叉历史问题

这套链式模型最现实的价值，就是处理 branching history。

想象一下：

1. 用户先让模型做规划
2. 模型读了一堆文件
3. 对话被 rewind
4. 换了另一条实现路线

如果只存一个平数组，结果通常只有两种：

- 旧分支直接丢了
- 或者旧分支混进现行历史里，很难区分当前有效线程

Claude Code 的链式 transcript 允许它：

- 保留真实发生过的分支
- 同时在加载时只重建当前活跃链

对一个工具型代理来说，这个能力非常实际，因为探索死路本来就是常态。

---

## 3. 会话文件不会一启动就落盘

存储子系统里一个很成熟的小设计是：

**session file 不会因为运行时启动就立刻创建。**

系统会先把一些 entry 暂存在内存里，直到出现真正的 user / assistant 消息，才物化成 JSONL 文件。

这避免了“幽灵会话”：

- 工具启动了
- 用户没真正用就退出
- 磁盘上却堆了一堆空会话文件

这是典型的规模化使用后才会在意的问题。终端助手开开关关很多次之后，存储卫生会变得很重要。

---

## 4. compact 不只是省 token，它会改变历史的语义

很多人容易把 compact 理解成：

“把前面对话压短一点。”

在 Claude Code 里，这个理解不够。compact 会直接改变“历史在存储层里的形态和含义”。

compact 之后，系统不希望 resume 时又把 pre-compact 的全部历史当成普通活跃上下文直接捞回来。

所以 Claude Code 引入了明确的 compact boundary 以及相关 metadata。

这些信息可能包括：

- preserved segment 相关指针
- boundary marker
- 和旧链条的逻辑关系
- 未来恢复时需要的状态线索

这说明 compact 不是删除，而是**替换 + 延续**。

系统需要记住：

- compact 发生过
- 什么内容被直接保留
- 什么内容被摘要替代
- 后续活跃线程应该从哪里继续

如果没有这些标记，compact 后的 resume 很容易重新hydrate错状态。

---

## 5. compact boundary 是语义事件，不是普通系统消息

compact boundary 最好理解成一种“状态切换事件”，而不是一条普通消息。

它意味着：

- 旧历史已经被转换过
- 当前之后的会话，要以摘要或 session memory 代替完整旧上下文
- 某些缓存和跟踪状态要清理
- SDK / CLI / transcript 层都需要知道这里发生了一次结构性变化

所以 compact boundary 会同时影响：

- transcript 持久化
- runtime state
- compaction metadata
- SDK 消息适配
- cleanup 与 cache invalidation

如果把它当“顺手插一条标记文本”，整个系统就会逐渐和真实状态脱节。

---

## 6. resume 之所以难，是因为 persistence 和 compaction 已经耦合在一起

如果会话历史很简单，resume 也会很简单。  
Claude Code 的历史并不简单，因为它同时支持：

- 流式工具交互
- 带状态的 session metadata
- sidechain
- compaction
- memory extraction
- fork session

所以 resume 绝不是“取最后 N 条消息”。

它真正要恢复的是：

- 当前活跃 conversation chain
- 当前 session identity
- 哪条分支才是 authoritative
- compact 之后的上下文语义
- post-compact 状态应该长什么样

这就是为什么 persistence 和 compaction 在代码里耦合得很深。  
一个不理解 compact semantics 的 resume 系统，最终一定会恢复错状态。

---

## 7. sidechain 的价值在于：保留真实历史，但不污染当前线程

sidechain 是整个存储子系统里我很喜欢的设计之一。

它让 Claude Code 同时拥有两种能力：

- 保留那些曾经真实发生过的分支历史
- 在加载当前会话时，又不会把这些旧分支混进活跃线程

对代理产品来说，这很重要。真实工作流本来就乱：

- 用户会改方向
- 模型会走死路
- 会有一些试过但放弃的路径

系统需要承认这些历史存在，但又不能让它们破坏当前连续性。

sidechain 就是这个折中。

---

## 8. compaction 其实同时也是持久化问题

很容易把 compact 只归到 context management。  
但在 Claude Code 里，compact 同样是一个 persistence 问题。

一旦 session 被 compact，存储层必须继续回答这些问题：

- 原来的链里哪些部分还直接存在？
- 哪些已经被摘要掉？
- 未来 resume 需要什么 metadata？
- 哪些旧消息已经不该再被视作活跃状态的一部分？

这也是 Claude Code 架构成熟的地方：  
它显然知道“上下文”和“存储”不是两张皮。

你怎么压缩历史，会直接决定你之后怎么存、怎么读、怎么恢复。

---

## 9. resume、continue、fork 其实是不同的连续性操作

从用户视角看，“继续之前的会话”可能像一件事。  
但从内部看，它至少分成几类：

- resume 原 session
- continue 既有历史
- fork 成新的 session identity

这几种操作并不等价。

只要系统同时支持这三种，它就必须非常小心：

- session ID
- parent 关系
- metadata 继承
- 哪些上下文应该沿用，哪些应该重写

所以你会看到存储层有些流程会重新 stamp metadata。  
因为消息虽然来自旧上下文，但现在要被纳入新的 session lineage。

这很细，但很必要。

---

## 10. 这套子系统的重要性，其实比表面看上去高得多

用户评价 AI coding 产品时，最常聊的是：

- 模型强不强
- 工具多不多
- 延迟快不快
- UI 顺不顺

但一旦进入真实工作流，“连续性”会变得同样重要。

如果一个系统不能可靠地：

- 停下
- 恢复
- compact
- 保住工作状态
- 在 branching 之后仍然连贯

那不管模型多强，都会显得很脆。

Claude Code 的 transcript + compact boundary 架构，就是为了避免这种脆弱感。

它是产品“耐用性”的组成部分。

---

## 11. 总体判断

Claude Code 的 transcript 子系统不是过度设计，而是匹配了它的真实使用场景：

- 长会话
- 工具密集
- 状态重
- 会分叉
- 会压缩
- 会恢复

它最重要的几个思想是：

- 历史是有结构的，不只是 append
- 分叉是常态，不是异常
- compact 会改变之后历史的语义
- resume 必须理解存储语义，而不只是消息文本
- 连续性本身就是核心产品属性

所以 transcript storage、compaction metadata 和 resume 逻辑才会在代码里绑得这么紧。

如果这是个一次性助手，这套机制会显得太重；但对一个认真的 coding agent 来说，这正是该有的机械结构。
