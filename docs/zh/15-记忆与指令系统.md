# Claude Code 的记忆与指令系统：它不是一种 memory，而是好几层

Claude Code 并不是只有一种“记忆”机制。它实际上同时使用了几层持久化上下文，而且每一层解决的问题都不一样。

这一点非常重要。一个生产级 coding agent 至少要同时满足三类需求：

- 有稳定的用户/项目指令
- 能在长会话里保持连续性
- 在上下文压缩时还能保住关键信息

Claude Code 对应的做法是：

- CLAUDE.md 指令层
- 基于 `MEMORY.md` 的文件型记忆层
- session memory 提取与压缩层

这篇就专门把这三层拆开看。

---

## 1. 先分清：不是所有持久化内容都叫 memory

读这套代码时最容易犯的错之一，就是把所有持久化上下文都笼统叫“记忆”。  
Claude Code 的架构其实分得更细。

至少可以拆成三类：

### 1. 指令型记忆

例如：

- `CLAUDE.md`
- `.claude/CLAUDE.md`
- `.claude/rules/*.md`
- `CLAUDE.local.md`
- managed instruction 文件

这类文件不是会话日志，而是告诉模型“以后该怎么做事”的规则层。

### 2. 持久工作记忆

也就是围绕 `MEMORY.md` 和专题 memory 文件的那一层。  
它更像项目/用户长期事实存储。

### 3. Session memory

这层更偏运行时，用来帮助长会话连续性和 compaction。

它更接近“持续演化的会话摘要”，而不是稳定的长期规则库。

从架构上看，这三类东西解决的是完全不同的问题，不能混成一个概念。

---

## 2. CLAUDE.md 本质上是一套指令覆盖系统

CLAUDE.md 绝不是“读一个提示词文件”这么简单。  
它更像一套分层的 instruction overlay system。

Claude Code 会按不同优先级去找：

- 机器级 managed instructions
- 用户级 instructions
- 项目级 instructions
- 本地私有 instructions

这意味着它允许行为塑形同时来自多个作用域：

- 组织策略
- 用户偏好
- 仓库规范
- 私有本地覆盖

这很现实，因为真正的开发环境往往就是多层规范叠在一起的。

### 为什么分层重要

这套层级让 Claude Code 能根据：

- 运行机器
- 当前用户
- 所在仓库
- 私有本地环境

表现出不同的行为模式。

所以 CLAUDE.md 更像行为配置系统，而不是单纯“提示词附件”。

---

## 3. include、rules 和 path 作用域让它更像配置基础设施

CLAUDE.md 这一层支持的能力包括：

- include 指令
- rules 文件
- 按路径生效的规则
- HTML comment stripping
- exclusion filter

这说明 Anthropic 要的不是“一个大文本框”，而是：

- 可模块化
- 可复用
- 可分作用域
- 可在本地精细注入

做到这一步时，这套系统已经更像“模型行为配置基础设施”，而不只是 prompt 文本。

---

## 4. CLAUDE.md 之所以重要，是因为 prompt 装配层给了它真正权力

只有当 CLAUDE.md 被带着强语气注入系统提示词时，它才真正有意义。

源码里的措辞很明确：这些规则应该覆盖默认行为指导。

这会直接影响模型在下面这些事情上的表现：

- 代码风格
- 工作流偏好
- 仓库约束
- 团队规范
- “在这里该怎么工作”的规则

这一层非常实用。很多团队真正在意的不是模型有没有多聪明，而是它在自己仓库里是不是“按这个地方的规矩干活”。

CLAUDE.md 就是把通用代理行为接到本地工作规范上的桥。

---

## 5. `MEMORY.md` 那套文件型记忆，解决的是另一类问题

除了指令层，Claude Code 还有自动记忆目录系统，以 `MEMORY.md` 为入口。

这层不是为了告诉模型“如何表现”，而是为了在跨会话时保留有价值的长期事实。

这里最好的设计点之一，是：

`MEMORY.md` 被当成一个索引，而不是无限笔记本。

推荐模式是：

1. 具体内容写到主题文件
2. `MEMORY.md` 只做轻量索引

这个选择非常正确。

如果总入口文件无限长大，它迟早会把 prompt 预算吃光。把 `MEMORY.md` 设计成索引，既保留了可导航性，又避免 always-loaded 内容失控。

---

## 6. 硬上限不是补丁，而是架构的一部分

Claude Code 对 memory entrypoint 有明确行数和字节限制。

这说明它已经非常清楚 memory 系统最常见的失败方式：

- 用户或模型不断追加记忆
- 记忆越来越肥
- 每次都要加载的内容越来越贵
- 真正需要做的会话工作反而被挤掉

所以系统里会有：

- 上限
- 截断
- 警告

这完全正确。

长时记忆系统的问题通常不是“记不住”，而是“什么都想记，最后把上下文拖死”。

Claude Code 的 memory 设计显然已经踩过这种坑。

---

## 7. Session memory 的存在，是因为反复全量总结太贵

Session memory 最有意思的地方在于，它和 compaction 紧紧绑在一起。

如果每次上下文压力上来，系统都要从零重新总结整段会话，那代价会很高：

- 延迟高
- 成本高
- 信息反复损失
- 压缩本身更容易失败

Session memory 提供了一条更可扩展的路：

- 平时增量保留重要事实
- compact 时用这部分记忆替换旧历史

从架构上看，这就把 compaction 从“每次重做全量摘要”变成了：

“近期消息 + 已维护好的 session memory”

这对长会话代理是明显更好的形态。

---

## 8. 指令型 memory 和 session memory 必须隔开

源码里一个非常重要但不容易一句话看出的边界是：

- 什么内容应该长期塑形未来行为
- 什么内容只是为了保住会话连续性

如果把这两类东西混在一起，会出很多问题：

- 临时任务细节被当成永久规则
- 本地偏好文件被塞满会话垃圾
- 模型分不清“怎么做事”和“发生过什么”

Claude Code 之所以更成熟，就在于它保住了这个边界：

- instruction files 负责行为塑形
- memory files 负责长期事实
- session memory 负责 continuity 和 compaction

这比很多 agent 系统要更有纪律。

---

## 9. Memory 系统同时也是成本与上下文策略

Claude Code 的 memory 架构不只是为“记得住”服务，也是在做资源分配。

一个长会话 coding agent 必须回答这些问题：

- 什么应该每次都进 prompt
- 什么只在需要时再读
- 什么应该摘要化
- 什么应该做索引
- 什么应该丢掉

Claude Code 的答案是结构化的：

- always-loaded instruction files 要有边界
- always-loaded memory entrypoint 也要有边界
- 详细记忆放到独立文件
- session memory 在 compaction 时替代旧历史

这样，memory 就不只是持久化功能，而是上下文资源策略的一部分。

---

## 10. 整套系统真正追求的是“行为连续性”

如果往上看一层，记忆与指令系统服务的其实是一个更大的目标：

**Claude Code 要在时间上持续表现得像同一个会工作的系统。**

这包括：

- 记得在一个仓库里该怎么做事
- 保留重要用户偏好
- 延续关键会话事实
- compact 之后仍然有上下文
- 中断恢复后还能接着干

这更接近“行为连续性”，而不只是字面意义上的记忆。

因为用户真正感知到的，不是磁盘上有没有一个文件，而是：

重新回来时，这个系统还懂不懂当前任务、当前仓库和当前工作方式。

---

## 11. 总体判断

Claude Code 的记忆架构，比“存点笔记以后再看”成熟得多。

它清楚地区分了：

- 指令
- 长期记忆
- 会话连续性机制

它在该有限制的地方加了硬上限。

它也把 memory 直接接进了：

- prompt 装配
- compaction
- 长会话行为
- 仓库级工作模式

所以在 Claude Code 里，memory 不是附属便利功能，而是架构主件之一。
